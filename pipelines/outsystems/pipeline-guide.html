OS → Mendix Migration Pipeline
Step-by-step guide for developers and migration consultants

What this pipeline does

Takes raw OutSystems 11 eSpace XML files — which contain the full application stack (screens, entities, logics, timers, APIs) — and transforms them into structured BRD scaffolds per module. These scaffolds drive Mendix development, replacing weeks of manual analysis.

The pipeline is source-agnostic by design. You can add extractors for other source types (C#, Java, Angular, SQL) and new mappers for additional Mendix concepts without changing the core engine.

Key design decisions

  • Phase 2 always processes ALL XML files — never a subset. The linker resolves cross-module references: if module A is missing from the corpus, module B's FK to an entity in module A becomes an unresolvable gap. Running a subset produces false gaps. The KB is always full-corpus (all 114 modules); Phase 3 filters per module internally.
  • Extractors write to extracted/, never knowledge-base/ — the merger owns the KB. This keeps extraction and merging cleanly separated so you can add new extractors without touching the merge step.
  • Mappers are pure functions — no file I/O, no globals. They receive items for one module and return an array. Easy to test, easy to add.
  • Use-case narrative is never fabricated — the scaffold marks every narrative field as an explicit TODO. Actors, preconditions, and main flow require business input.
  • Gap = unresolved cross-reference — when a FK target or action reference can't be resolved, a gap string is added. Gaps roll up to confidence scoring. Not all gaps indicate real problems — see the Gaps & Noise section.
Phase 1 — Source Analysis
Understand what you have before running any scripts — this phase is human + AI judgment, no commands

What is this phase?

Before touching the pipeline, you need to understand your source. Phase 1 is not a script — it is a manual review step where you (with AI assistance) open the source files, identify the platform, count the modules, and classify them into tiers. This takes 15–30 minutes and prevents running extraction on the wrong files or in the wrong format.

Step 1 — Identify the platform

Open one of the source files. The root XML tag tells you the platform:

Root tag / file signaturePlatformExtractor to use
<ESpace Version="11">OutSystems 11xml-extractor.js
pom.xml + @Entity annotationsJava / Spring Bootjava-extractor.js (add)
*.csproj + DbContext.NET / C#cs-extractor.js (add)
*.fmb / *.fmxOracle Forms(contact team)
This project uses OutSystems 11 eSpace XML files. The XML contains the full application stack per module: entities (data), logics (business rules), screens (UI), timers (scheduled jobs), and API definitions — all in one file.

Step 2 — Count and classify the modules

List the source files and group them into three tiers:

Step 3 — Optional: validate the extractor on a sample

Run this before Phase 2 if you are doing this for the first time or after changing the XML source:

node run.js 1 xml

Output: pipeline/samples/schema.json — a small preview of what the extractor will produce. Check that entity attributes, logic kinds, and screen summaries look correct. If something looks wrong, fix extractors/xml-extractor.js now — it is much faster to iterate on 8 files than 114.

Do not skip Phase 1. Running extraction on the wrong files, or on a platform the extractor doesn't understand, produces silent bad output that's hard to debug downstream.
Phase 2 — Extraction
Read all 114 OS XML files and produce a structured JSON knowledge base — the foundation for everything that follows

What happens here?

The extractor reads every XML file, pulls out all the application artifacts (entities, actions, screens, timers, APIs), and writes them to structured JSON files in knowledge-base/. After extraction, the linker stitches cross-module references together — for example, connecting a foreign key in module A to the entity it points to in module B. This JSON knowledge base is what all downstream steps (BRD generation, HTML report) read from.

You never edit these JSON files manually — they are always regenerated from source.

Configuration first

Set paths in pipeline/config.json before running:

{
  "blueprintDir": "C:/path/to/OS-ExtractedXML",
  "shareDir":     "C:/path/to/Share/docs",
  "dbDir":        "",
  "docsDir":      "C:/path/to/Share/design-specs"
}

Command

node run.js 2 xml        # XML only (most common — covers full OS stack)
node run.js 2            # all configured source types

What the linker does

After extraction, lib/linker.js runs cross-reference rules:

RuleWhat it links
X1Entity FK attribute → referenced Entity (by name)
X2Logic ExecuteAction → target Logic (by uniqueId key)
X3Screen permission → Role (by uniqueId key)
X4Timer → Logic (by actionRef key)
X5Screen clientActions → Logic
X6ServiceAPI → Logic (same-module name match)

Unresolvable references become gaps — see the Gaps & Noise section.

Phase 3 — BRD Scaffold Generation
Transform the KB JSON into structured BRD files, one per OS module

Purpose

Runs five mappers per module against the knowledge base and produces one .brd.json file per OS module. These are the primary handoff artifact from automated analysis to human review.

Command

node run.js 3

Output: knowledge-base/brd/{ModuleName}.brd.json (one per module)

Also writes: knowledge-base/brd/generation-report.json (counts + warnings)

The five mappers

MapperInputOutput sectionKey logic
domain-entity-mapper entities + staticEntities domainEntities[] isStatic → Enumeration; else PersistentEntity. Extracts key attributes (mandatory + FK only).
microflow-mapper logics microflows[] Name-pattern rules: GET_* → retrieval, ACT_Save* → write, VAL_* → validate, etc.
page-mapper screens + webBlocks pages[] + webBlocks[] widgetSummary → UI pattern: list+form, list, form, or detail.
use-case-mapper screens useCases[] Scaffold only — all narrative fields are explicit TODOs. No fabrication.
integration-mapper serviceApis + extEntities integrations[] serviceApis → inbound; extEntities → outbound. Empty array is valid.

Phase 4 — Report & Business Review
Understand the extraction results and review BRD scaffolds with stakeholders

Purpose

Generates a self-contained interactive HTML report from the KB JSON and BRD files. This is the primary review surface — everything a business analyst or developer needs to understand the OS application and plan the Mendix rebuild.

Command

node generate-report.js
# → knowledge-base/extraction-report.html

Open the HTML file in any modern browser (Edge, Chrome). No server needed — fully self-contained.

What the report shows

SectionWhat to look at
DashboardTotal counts, BRD coverage %, open gaps

Module — BRD sectionAuto-generated entities, microflows, pages, use case scaffolds

Business review workflow

  • Open the report. Start with the gap heatmap — low-confidence modules need most attention.
  • Pick a priority module. Click it in the sidebar. Check its BRD summary section.
  • Review use case scaffolds. Each TODO field needs business input: actors, preconditions, main flow, postconditions.
  • Fill in with Claude. Open mxcli-project-toolkit/skills/brd-generation.md Step 3 prompt template. Paste the scaffold + KB context and ask Claude to produce a narrative BRD.
  • Resolve open gaps. Cross-module references marked as gaps may need business decisions about ownership or shared entities.
  • Sign off the BRD. Once reviewed, change "reviewStatus": "approved" in the BRD JSON. This feeds into Phase 5.
Glossary
Key terms used throughout this pipeline and guide

BRD — Business Requirements Document

The primary handoff artifact from automated analysis to Mendix development. One BRD file (.brd.json) is generated per OutSystems module. It contains auto-extracted sections (domain entities, microflows, pages, integrations) plus use-case scaffolds with explicit TODO fields that must be filled in by the business team. BRDs drive Mendix development; they replace weeks of manual module analysis.

Knowledge Base (KB) — the JSON corpus

The set of JSON files in knowledge-base/ that represent all extracted and cross-referenced data from the OS source files. The KB is always full-corpus (all modules), fully merged, and enriched with cross-reference links and gaps by the linker. It is the foundation that all BRD mappers and the HTML report read from. Files include entities.json, logics.json, screens.json, and others.

The KB is generated — never edit it manually. Re-run Phase 2 to regenerate from source.

eSpace

An OutSystems module. Each eSpace is exported as one XML file (.oml format). It contains the full application stack for that module: entities, logics (server/client actions), screens, timers, roles, and API definitions. This project has 114 eSpace XML files.

merger.js

The library that runs after the extractors in Phase 2. It takes the raw extractor output from extracted/ (one JSON file per extractor), deduplicates items by linkId, and splits them by type into the typed KB files (entities.json, logics.json, etc.). If you add a second extractor (e.g. Java or SQL), the merger combines both sources transparently.

linker.js

Runs cross-reference resolution across all KB items. It applies six rules (X1–X6) that match references between item types — for example, an entity FK attribute is matched to the entity it references, or a timer is matched to the logic it fires. Resolved references are added as _links on the item. References that cannot be resolved become gaps.

Gap

A gap is a cross-reference that the linker could not resolve — for example, an entity FK pointing to an entity name that doesn't exist in the KB, or a timer referencing a logic key that wasn't found. Gaps are stored as strings on the item (e.g. fk-unresolved:User) and roll up into a confidence score for each module in the BRD and report.

Not all gaps indicate real design problems. Many are expected noise from OS platform system entities (User, Group, Role) that have no XML representation. See the Gaps & Noise section for how to handle each type.

Gaps & Noise Reduction
Understanding what gaps mean and which ones to suppress

What a gap is

A gap is an unresolved cross-reference — something the extractor found a reference to but couldn't match to a known item in the KB. Gaps affect the module's confidence score and appear in the extraction report.

Not all gaps indicate missing design. Some are expected noise that should be suppressed.

Gap types and how to handle them

Gap typeCauseAction
no-db-table-found OS XML doesn't expose the underlying SQL table name (ossys_EntityName) Suppress when no DB source is provided — irrelevant for Mendix migration
fk-unresolved:User FK to OS platform User entity (not in XML) Map to Administration.Account in Mendix
fk-unresolved:Group FK to OS platform Group entity Map to Administration.UserRole
fk-unresolved:Espace OS internal application-space reference Ignore — no Mendix equivalent
fk-unresolved:Role FK to OS platform Role entity Map to Administration.UserRole
fk-unresolved:OtherEntity FK to an entity in a different module not yet extracted Review — may indicate a cross-module dependency to design
timer-action-unresolved Timer references a logic key that wasn't found Review — action may be in a different module or removed

How to add a suppression rule

In pipeline/lib/linker.js, add to the PLATFORM_ENTITIES constant:

const PLATFORM_ENTITIES = new Set([
  'user', 'group', 'espace', 'role'
  // add new system entity names here (lowercase)
]);

// In Rule X1:
if (!target) {
  const refLower = attr.referencedEntity.toLowerCase();
  if (!PLATFORM_ENTITIES.has(refLower)) {
    map[entity.linkId].gaps.push(`fk-unresolved:${attr.referencedEntity}`);
  }
}

For no-db-table-found, the fix is to only fire the gap when a DB source was actually provided:

// Rule 1: only flag if we actually have DB data
if (tables.length > 0) {
  const match = tables.find(t => t.name.toLowerCase() === expectedTable.toLowerCase());
  if (!match) map[entity.linkId].gaps.push('no-db-table-found');
}
Adding a New Extractor
How to add support for a new source code type (Java, C#, Angular, SQL, …)
See pipeline/extractors/README.md for the full detailed guide. This page covers the concepts.

When to add a new extractor

Add an extractor when you have source code in a format that contains design information not available in the OS XML — for example:

  • Java backend (Spring Boot) — business logic, REST controllers, JPA entities
  • Angular frontend — components, services, routing
  • SQL scripts — table DDL, stored procedures, views
  • C# backend (ASP.NET) — controllers, models, EF entities

The four steps

  • Create the extractor — pipeline/extractors/{type}-extractor.js. Parse your source files. Push normalized items following the item interface to an array. Write the array to pipeline/extracted/{type}.json.
  • Create a sampler — pipeline/samplers/{type}-sampler.js. Reads N files and outputs a schema summary to pipeline/samples/{type}-schema.json.
  • Wire into run.js — add the sampler to phase1() and the extractor to phase2(). Add the source path to config.json.
  • Add linker rules (optional) — if your new type has cross-references to OS entities, screens, or logics, add matching rules in lib/linker.js.

Item interface (required fields)

{
  "type":     "entity | logic | screen | ...",  // or a new type you define
  "linkId":   "java:class:MyService",           // globally unique
  "uniqueId": "java:class:MyService",           // same or more specific
  "name":     "MyService",
  "module":   "ModuleName",                     // match OS module name if applicable
  "_source":  "/path/to/source/file",
  "_gaps":    [],   // leave empty — linker populates this
  "_links":   []    // leave empty — linker populates this
}

Example: Java extractor pattern

'use strict';
const fs   = require('fs');
const path = require('path');

const sourceDir  = process.argv[2];
const outputFile = path.join(__dirname, '..', 'extracted', 'java.json');

const items = [];

// Walk .java files
for (const file of walkDir(sourceDir, '.java')) {
  const src = fs.readFileSync(file, 'utf8');

  // Parse @RestController classes
  if (src.includes('@RestController')) {
    const name = extractClassName(src);
    items.push({
      type: 'logic',
      linkId: `java:controller:${name}`,
      uniqueId: `java:controller:${name}`,
      name, module: inferModule(file),
      logicKind: 'action',
      isPublic: true,
      _source: file, _gaps: [], _links: []
    });
  }
}

fs.writeFileSync(outputFile, JSON.stringify(items, null, 2), 'utf8');
console.log(`Extracted ${items.length} Java items`);
Adding a New BRD Mapper
How to add a new section to the BRD scaffold (roles, workflows, structures, …)
See pipeline/generators/brd-mappers/README.md for the full guide.

When to add a new mapper

Add a mapper when:

  • There's a KB item type that isn't yet represented in the BRD output
  • You want a new Mendix concept section (e.g. Workflows, Security, Non-persistent Entities)
  • You added a new extractor and want its data to appear in the BRD

The three steps

  • Create the mapper — pipeline/generators/brd-mappers/{name}-mapper.js. Export a map(moduleItems, allItems) function that returns an array. Add _gaps[] to each item for unresolvable references.
  • Wire into index.js — import your mapper and call it inside mapModule(). Add the result to the BRD JSON object. Update the summary count. Spread gaps into openGaps.
  • Update the report — add a display table for your new section in generate-report.js inside the showModule() JS function.

Mapper template

'use strict';

function map(moduleItems, allItems) {
  const results = [];

  for (const item of (moduleItems.roles || [])) {
    const gaps = [];

    // your mapping logic here

    results.push({
      name:        item.name,
      module:      item.module,
      mendixType:  'UserRole',
      description: item.description || '',
      _gaps:       gaps,
    });
  }

  return results;
}

module.exports = { map };

Deferred: Workflow mapper

BPT processes (logicKind: 'process') are already extracted into logics.json and appear as bptProcessCount in every BRD. A workflow-mapper.js is the natural next mapper to build.

Before building it, agree with the business on:

  • Approval chain structure — single route or branching?
  • Escalation rules and SLA timers per step
  • Which roles participate at each step
  • How rejections / remands work
Once designed, the mapper produces a workflows[] section with steps, transitions, role assignments, and SLA configuration — ready for the Mendix Workflow editor.
Adapting the Pipeline for a New Source Platform
How to migrate a Java/Angular, SAP, or other stack using the same engine

The pipeline is platform-agnostic by design

The core engine (merger, linker, BRD mappers, report generator) works on normalized KB JSON. Only the extractors are platform-specific. To migrate a different source platform, you only need to write new extractors that emit the same item interface.

What to change for a new platform

ComponentChange needed
extractors/Write new extractor(s) for the source platform. The OS xml-extractor.js is a good reference.
config.jsonAdd new source path keys for the new platform.
run.jsAdd new extractor to phase1 and phase2 blocks.
lib/linker.jsAdd cross-reference rules for the new platform's linking patterns.
brd-mappers/Reuse existing mappers if item types match. Add new mappers for platform-specific concepts.
generate-report.jsUsually no change needed — the report reads from the KB generically.
mxcli-project-toolkit/skills/Add a new source-{platform}.md skill file documenting the source schema.

Example: Java/Spring Boot + Angular migration

  • Java extractor — parse @Entity classes → entity items. Parse @RestController methods → logic/action items. Parse @Service classes → logic/action items.
  • Angular extractor — parse component classes → screen items. Parse service calls in templates → link to backend logics.
  • SQL extractor — parse table DDL → enrich entity items with physical schema. Parse stored procedures → logic/action items with SQL body.
  • Concept mapping — Spring @Entity → Mendix Persistent Entity. REST controller method → Microflow. Angular component → Page. @Scheduled → Scheduled Event.
  • BRD mappers — existing domain-entity, microflow, page, integration mappers should work without changes if the item interface is followed. Adjust purpose-inference patterns in microflow-mapper.js for Java naming conventions.

Cross-platform projects

If a Mendix module needs data from both OS XML and, say, a Java backend — run both extractors in Phase 2. The merger deduplicates by linkId and the linker cross-references across all item types. The BRD per module will include items from both sources.

This is the long-term vision: one pipeline, many sources, one Mendix BRD per domain.
