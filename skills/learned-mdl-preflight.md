# MDL Pre-flight Checklist — STOP conditions before writing any script
**Applies to:** any mxcli project.
**Purpose:** Before drafting any MDL script, check every planned operation against this table. Each STOP row was born from a real MPR corruption or silent runtime failure — not theoretical concerns. Skipping this check has caused project-unloadable corruption every time.

**Source:** Generalized from a live Mendix 11.12.0 Beta project, 2026-07-06/07. Incident detail stays in that project's own `bug-logs/` — only the generalizable rule is here.

**Version context:** Rules marked "STOP" were confirmed on specific mxcli/Mendix versions noted per entry. Rules 1b and 10 were retested 2026-07-09 on mxcli v0.13.0 / Mendix 11.12.0 and confirmed resolved. Rule 9 was retested 2026-07-09 via `mxcli --mcp` on v0.13.0 and confirmed resolved (BSON serialization bug only affects the disk write path; `--mcp` bypasses it entirely). If in doubt, retest on your version and stamp the result in `bug-logs/mxcli-bugs.md`.

## Step 0 — choose the write mode (do this first, before the STOP table)

There are three write modes, and they are **co-equal tools chosen by the shape of the work** — not one default plus two fallbacks. Pick the mode from the task first; *then* use the STOP table below only as the safety overlay that overrides your pick for the handful of operations mxcli corrupts. "CLI unless forced to MCP" is the wrong mental model — it is why projects drift to CLI-only and never touch the live-editing loop.

| Mode | Command | Reach for it when | SP |
|---|---|---|---|
| **CLI** (disk write) | `./mxcli exec script.mdl` | Bulk, structural, done-once: a whole domain model, a module's microflows, access rules, navigation, demo users. One script scaffolds a lot. | Closed |
| **MCP + MDL** | `./mxcli --mcp http://localhost/mcp --mcp-dial localhost:7782 exec script.mdl` | Iterative refinement and UI tuning where restarting SP per change would kill flow — page layout, widget wiring, visibility expressions. Live edit, no recompile wait. Also the required path for rule 9 (inline assoc-sets). | Open |
| **Hand-rolled MCP** | `pg_patch_page`, `ped_create_document`, `ped_update_document` | Widget JSON shapes MDL has no syntax for — rules 1, 6, 6b, 7 and datagrid customContent conditionals. Confirmed payloads in `learned-mcp-patterns.md`. | Open |

**Typical module rhythm:** one CLI exec to scaffold domain + microflows → MCP+MDL for page iteration and UI tweaks → hand-rolled MCP only for the specific widget shapes MDL can't reach. A project that only ever runs CLI is usually leaving the fast UI-iteration loop on the table, not being more disciplined.

**When more than one mode genuinely works** — most page changes and most single/small microflow builds fall in this zone — don't silently pick one. Name the viable modes (CLI / MCP+MDL / hand-rolled MCP — remember MCP splits into these two: MDL run through the MCP connection, or direct `pg_patch_page`/`ped_*` calls with no MDL at all), give a one-line recommendation with reason (e.g. "MCP+MDL, since SP is already open and this is a small widget tweak" / "CLI, this is a whole-module scaffold"), and let the user pick before executing. Only skip asking when the STOP table forces a mode, or the user has already stated a standing preference for that exact operation type in this session/project. Default lean, absent other signal: MCP for page edits and small microflow builds, CLI for bulk/structural work (domain models, access rules, navigation, demo data).

Once you've picked a mode per operation, run the STOP table below against every one of them — it overrides your pick whenever an operation falls in the corrupting set.

**Companion skills:** `learned-mcp-patterns.md` (MCP alternatives), `bug-logs/mxcli-bugs.md` (bug detail), `iterative-build-loop.md` (exec discipline), `learned-microflow-patterns.md` (annotation + inline assoc rules)

## The checklist — run before writing a single line of MDL

| # | If your script will… | Then… | Root cause |
|---|---|---|---|
| 1 | Use `visible:` or `editable:` **conditional expressions** on containers **inside a `datagrid customContent` column** | **STOP → MCP** (`pg_patch_page`) | mxcli writes blank `AttributeIdentifier` → `StorageLoadException` on SP open. **BUG-18, still open on v0.13.0** (found 2026-07-03 after v0.13.0 release). **Primary real-world trigger: per-value badge pattern** — multiple containers with `Visible: [State = ENUM.Value]` inside a column. mxbuild passes clean; SP load fails. Safe CLI alternative: `dynamictext (Attribute: "State")` renders enum caption natively. Do NOT use `ContentParams: [{1} = toString("State")]` inside customContent — resolves as attribute path → CE1613. Per-value colours require `pg_patch_page` MCP. |
| 1b | Use `visible:` or `editable:` conditional expressions on **regular widgets / dataview / page-level containers** in a `CREATE PAGE` or `CREATE SNIPPET` statement | Safe on mxcli v0.13.0 — retested 2026-07-09 on WMS-LargeSource (Mendix 11.12.0), 0 mxbuild errors. Fixed by codec engine (#627). **Use mxcli.** | Was: STOP in pre-v0.13.0; no longer applies to regular widgets in CREATE. |
| 1c | Use `ALTER PAGE { SET Visible = '...' ON widgetName }` to add/change a **conditional visibility expression** on an existing widget | **STOP → Studio Pro GUI** | `ALTER PAGE SET Visible` writes a blank `AttributeIdentifier` on the `ConditionalVisibilitySettings` → `StorageLoadException` on SP open. Confirmed 2026-07-20 on Mendix 11.12.0 / mxcli v0.13.0 (WMS-App). Note: Rule 1b (CREATE PAGE with Visible) is safe; only the ALTER path is broken. **Recovery:** `./mxcli -p WMS-App.mpr -c "ALTER PAGE Mod.Page { SET Visible = '' ON widgetName }"` clears the bad value, then set the expression via SP. |
| 2 | Call `alter settings configuration`, `alter settings model`, or `alter project security level` | **STOP → Studio Pro GUI only** | Deterministic BSON stream-desync on the Settings unit — confirmed corrupt on every retry (see BUG-LOCAL-05) |
| 3 | Drop an attribute that has security grants applied to it | **STOP → Studio Pro GUI** | mxcli removes the attribute but leaves dead UUID pointers in the entity access rules → `KeyNotFoundException` on load (BUG-01) |
| 4 | Bind a widget to `AutoChangedBy` or `AutoChangedDate` system attributes | **STOP → omit or MCP** | Produces CE1613 dangling reference — these attributes are not bindable via mxcli widget expressions |
| 5 | Pass scalar (non-entity) parameters from a page button to a microflow | **STOP → redesign** | Mendix platform constraint: page buttons can only pass entity objects to microflows, never individual String/Integer/Decimal values. Silently fails to bind (CE1571). The microflow must take the whole object. |
| 6 | Use a COMBOBOX widget in association mode | **STOP → MCP** (`pg_patch_page`) | mxcli rejects the `Entity` property required by SP (MDL-WIDGET01), SP requires it (CE0642) — no working mxcli path exists |
| 6b | Use `DatagridDropdownFilter` in association (ref) mode as a DataGrid2 filter | **STOP → MCP** (`pg_patch_page`) | Confirmed working JSON pattern exists — see `learned-mcp-patterns.md`. `refOptions` microflow must be no-param; use `DomainModels$IndirectEntityRef` for `refEntity`; use `Pages$MicroflowSource` (not `CustomWidgets$CustomWidgetMicroflowSource`) |
| 7 | Use a cross-module association traversal as a widget datasource (`$currentObject/OtherModule.Assoc`) in DataGrid/DataView/ListView | **STOP → MCP** (`pg_patch_page`) | mxcli writes null `DestinationEntityId` on the `EntityRefStep` → `StorageLoadException` on SP open. Same-module traversals are fine. |
| 8 | Create any association — same- or cross-module | **STOP → run `SHOW ASSOCIATIONS` first** | No `IF NOT EXISTS` support — re-running a CREATE silently duplicates the association; mxbuild then flags CE0065/CE0069. Only write `CREATE ASSOCIATION` for names that do not yet appear in `SHOW ASSOCIATIONS` output. |
| 9 | Set an association inline in a CHANGE or CREATE activity: `change $Obj (Module.AssocName = $Other)` or `create Entity (Module.AssocName = $Other)` | **Depends on whether the association crosses modules — check this before picking a mode.** **Cross-module (e.g. to `System.User`, `Administration.Account`, or any other module's entity): STOP → `mxcli --mcp exec script.mdl`** (SP must be open) — confirmed still corrupting on **v0.16.0** as recently as 2026-07-13 (`change $NewAccount (System.User_UserRoles = $Role)` → invalid-`AttributeIdentifier` load failure, KT-POC project, two branches). **Same-module (both entities in the association live in the same module — including a marketplace module's own internal associations, not just a project-authored module): confirmed SAFE via plain CLI on v0.16.0**, 2026-07-23 (TFC-TCXGraphPOC) — an isolated throwaway test (2 new entities + 1 new association, all in one project module) and a real 176-activity production microflow (46 inline association-sets, all within the same module) both passed a full `mx check` clean. Do not assume this same-module finding extends to cross-module cases — those two are separately confirmed, in opposite directions, on the identical mxcli version. When genuinely unsure which category an association falls into, treat it as cross-module (the safer default) until confirmed otherwise. **⛔ Never resolve this STOP by dropping the association set** — the object still needs the link. Routing to `--mcp` (or MCP fallback) preserves it; silently omitting it ships a broken golden path (a real WMS incident: a scanned order created with no link to its unit/location, mxbuild clean). If the brief's golden-path data-effects table lists an association for this action, it must be set. | mxcli's disk-write serializer writes the association name as an `AttributeIdentifier` in CHANGE/CREATE BSON in a way that fails module-boundary resolution on SP load for cross-module associations specifically — same-module associations apparently resolve fine through the same code path. `--mcp` routes through SP's own engine, bypassing mxcli's serializer entirely, and is confirmed safe for both cases. **Reading through an association (`$Obj/Module.Assoc/Target/Attr`) is safe on any path, cross-module or not.** |
| 10 | Use `count($list)` inside a `declare` expression | Safe on mxcli v0.13.0 — retested 2026-07-09 on WMS-LargeSource (Mendix 11.12.0). Both `$x = count($list)` and `declare $x integer = count($list)` pass mxbuild with 0 errors. **Use mxcli.** | Was: STOP / CE0117 in pre-v0.13.0; no longer applies. **Caveat added 2026-07-20:** `count()` is not a valid Mendix expression function at all — Mendix's own function is `length($list)`. A `log info` activity using `toString(count($ActiveRoutes))` passed `mxcli check` but failed `mx check`/SP with CE0117. It is unclear whether the 2026-07-09 retest actually exercised a real `count()` call or a different construct; treat `count()` as unverified and prefer `length($list)` until re-confirmed. |
| 11 | Filter an XPath with an association-traversal as the comparand: `[Assoc_X = $Ctx/Other.Assoc]` | **STOP → retrieve target first, then filter** | XPath constraint right-hand side must be a directly-bound variable, not an association path. Pattern: `retrieve $Target from ... where [...]; retrieve $Result from Module.Entity where [Module.Assoc_X = $Target]` |
| 12 | Write `$currentObject/Attr` **inside** a `ContentParams` value on a `dynamictext` — bare or wrapped in a function, e.g. `ContentParams: [{1} = someFunction($currentObject/Attr, ...)]` or even `ContentParams: [{1} = $currentObject/Attr]` | **Drop the `$currentObject/` prefix — use the bare attribute name.** `ContentParams: [{1} = toString(Attr)]` or `ContentParams: [{1} = Attr]`. Function calls themselves are fine; the prefix is the trigger. | mxcli serializes the whole ContentParams value as an attribute-path suffix and concatenates the dataview's qualified entity name onto it — producing a nonsense path, CE1613: "The selected attribute '<Entity>.<literal expression text>' no longer exists." Confirmed on Mendix 11.12.0 / mxcli v0.13.0, 2026-07-16 and re-confirmed 2026-07-20 (root cause corrected — see BUG-23): the trigger is the literal `$currentObject/` prefix, not "any function call" as originally written here, and it fails even with NO function at all (bare `$currentObject/Attr`). **`Attribute: "AttrName"` is NOT a safe alternative for non-string types** — mxcli silently re-serializes it back into the same broken `ContentParams: [{1} = toString($currentObject/Attr)]` form (confirmed via `describe page`), so it round-trips into this exact bug. **Do not confuse with `DynamicClasses`/`Visible` expressions, which take the opposite form and genuinely require `$currentObject/Attr`** — e.g. `DynamicClasses: 'if $currentObject/State = Enum.Value then ...'` is correct and already the working pattern elsewhere in this codebase. `mxcli check`/`describe page` never catch this — only a real `mx check`/`docker check`/SP compile does. |
| 13 | Filter a `retrieve` by an association in a Retrieve object(s) activity: `retrieve $X from Module.Entity where AssocName = $Other` | **Fully module-qualify the association name in the WHERE clause**: `where "OwningModule"."AssocName" = $Other` | An unqualified association name in a retrieve XPath constraint compiles (`mxcli check` passes) and even loads in SP without a syntax error, but fails Studio Pro's own `mx check`/build with CE0161 "Error(s) in XPath constraint." Confirmed 2026-07-20 on Mendix 11.12.0 (WMS-App): two retrieves using the bare association name (`TransportOrder_TransportUnit`, `TransportOrder_TargetLocation`) both failed; a third retrieve in the same microflow that already used the fully qualified form (`Transportation.TransportOrder_TransportUnit`) passed clean. `mxcli check --references` does not catch this — only `mx check` / SP does. |
| 15 | Quote attribute names inside a **`CHANGE`** activity argument list: `change $Obj ("State" = val)` | **STOP → never quote attribute names in CHANGE argument lists** | mxcli writes an invalid `AttributeIdentifier` to BSON — `StorageLoadException` on SP open, project unloadable. Confirmed 2026-07-20 on Mendix 11.12.0 / mxcli v0.13.0 (WMS-App): `change $Order ("State" = $MappedState, "StartDate" = [%CurrentDateTime%])` corrupted the MPR, required full backup restore. `mxcli check` and `mxcli check --references` both pass — only `mx check` / SP open reveals the damage. **Correct form:** `change $Order (State = $MappedState, StartDate = [%CurrentDateTime%])` — never quote attribute names in CHANGE value/expression positions. **Scope narrowed 2026-07-23:** originally written to cover `CREATE` too, but TFC-TCXGraphPOC's `done-06-seed.mdl` (StyleGallery) has multiple `CREATE` activities with quoted attribute names (`"TFCId" = ...`, `"PartName" = ...`) that executed clean and gated `done-` (`mx check` 0 errors) — the confirmed corruption incident was CHANGE-specific, not CREATE. Quoting attribute names in `CREATE` argument lists is safe; only `CHANGE` is a STOP. |
| 14 | `revoke Module.Role on Module.Entity (write *)` followed by `grant Module.Role on Module.Entity (write (list))`, when another role has its own separate `grant` on the same entity | **STOP → revoke ALL roles' access on the entity first, then re-grant every role's full rule in one script** | mxcli's revoke/grant on one role can collapse or silently merge the entity's access-rule set, dropping a *different* role's untouched grant entirely. Confirmed 2026-07-20 (WMS-App): revoking `write *` and re-granting an explicit write list for `TransportOperator` caused `TransportViewer`'s separate read-only grant to disappear, and `TransportOperator` silently lost its `create`/`delete` structural rights — surfacing as ~48 new CE2729 "no read access" errors on unrelated pages. Root cause only visible via `DESCRIBE ENTITY` after the fact — `mxcli check --references` and even a subsequent `mx check` pass to reveal the fallout, not the mechanism. **Safe pattern:** `revoke Role1 on Entity; revoke Role2 on Entity; grant Role1 on Entity (...); grant Role2 on Entity (...);` — rebuild the whole entity's security from scratch in one script whenever any existing rule needs its write/read list changed, never patch one role's rule in isolation. |
| 17 | Write a compound boolean expression combining uppercase `AND`/`OR` with a `!=` comparison anywhere in the same expression — in a Decision/IF condition OR a `retrieve ... where` clause | **Use lowercase `and`/`or` for boolean keyword operators.** Confirmed safe in every combination tested: lowercase `and`/`or` + `!=`, uppercase `AND`/`OR` + only `=` comparisons. Only uppercase `AND`/`OR` + a `!=` in the same expression fails. Given the narrow, easy-to-miss trigger, default to lowercase `and`/`or` everywhere as a blanket-safe habit rather than trying to remember the exact boundary. | Confirmed 2026-07-23 (TFC-TCXGraphPOC, mxcli v0.16.0) via isolated tests: `if $Flag = Enum.HIGH AND $Outcome != Enum.DieGoHold then` → CE0117 "Error(s) in expression"; parentheses around each comparison did NOT fix it; switching to lowercase `and` did. Same bug reproduced independently in a `retrieve ... where PartId = 'x' AND SupplyPlant = $P AND "Status" != Enum.Archived` → CE0161 "Error(s) in XPath constraint"; lowercase `and` fixed it too. `mxcli check --references` does not catch this — only a real `mx check`/SP compile does. Root cause appears to be in mxcli's expression/XPath serializer mishandling the token sequence, not a real Mendix expression-language restriction (Studio Pro accepts lowercase `and` combined with `!=` without issue). |
| 16 | Write an access-rule XPath constraint with a **path expression inside the `[%CurrentUser/...%]` substitution token** — `[Assoc = '[%CurrentUser/Module.OtherAssoc%]']`, especially crossing module boundaries | **Never put a path inside the CurrentUser substitution. Keep it bare (`'[%CurrentUser%]'`) and do the ENTIRE multi-hop/cross-module traversal on the left-hand side instead**, qualifying the entity name at every module crossing (same rule as the "cross-module association expression paths" MDL gotcha, extended here to access-rule XPath constraints, not just microflow expressions). Was: `[FeasibilityDecision_Supplier = '[%CurrentUser/TFC.Account_Supplier%]']`. **Fix (confirmed working):** `[TFC.FeasibilityDecision_Supplier/TFC.Supplier/Administration.Account_Supplier = '[%CurrentUser%]']` — walk Supplier→Account fully on the left, compare directly to the (already-an-Account) CurrentUser token. | This is genuinely invalid Mendix XPath, not an mxcli-specific bug — `CE0161` "Error(s) in XPath constraint" is Mendix correctly rejecting it every time it's checked, which is why it kept resurfacing across three separate incidents on TFC-TCXGraphPOC (2026-07-22/23, `bug-logs/mxcli-bugs.md`) that were originally misdiagnosed as mxcli corrupting the domain model on unrelated grant rewrites. In fact the "unrelated entity broke too" symptom was just two independently-broken rules (both using the same invalid CurrentUser-path form) surfacing together once a full `mx check` ran — not one rule corrupting the other. Once rewritten to the correct bare-CurrentUser/full-LHS-chain form, the earlier "Studio Pro GUI only, module-wide" restriction no longer applies — **confirmed mxcli-safe 2026-07-23** via `test-05-currentuser-xpath-chain.mdl` (rebuilt all 4 role grants on the affected entity through mxcli exec, reproducing the corrected XPath verbatim, `mx check: 0 errors`). Use mxcli directly for this pattern; no GUI fallback needed. |

**Default to mxcli for:** entities/attributes/enums, associations (after SHOW ASSOCIATIONS check), microflows (without inline assoc-sets), demo users, module roles/grants, navigation.

## MDL gotchas (quick reference — common traps)

These don't require a STOP, but will cause silent failures or check errors if missed:

- **Never quote attribute paths on ANY variable, not just `$currentUser`:** write `$currentUser/Name` and `$Param/ResolutionNote`, never `$currentUser/"Name"` or `$Param/"ResolutionNote"`. The quoted form is syntactically valid MDL so `mxcli check` passes, but SP/`mx check` rejects it at load/build time with CE0117 ("Error(s) in expression"). Confirmed 2026-07-20 across `$currentUser/"Name"` (three TransportEvent CREATE activities) AND `$RetryInput/"ResolutionNote"` (a NPE parameter, two activities in a different microflow) — this is a general variable-attribute-traversal rule, not a `$currentUser`-specific quirk. Only quote the attribute when it conflicts with an MDL reserved word, never as a blanket safety measure.

- **Grants belong co-located with the element — never deferred to a single security script:** write `grant execute` at the end of the microflow script that creates the microflow; `grant view` at the end of the page script; entity access grants at the end of the domain script. A deferred `security.mdl` that does all grants at once is the primary cause of silently inaccessible pages and microflows — mxbuild produces no CE error for a missing grant, so the only signal is a blank screen at runtime. A `roles.mdl` script for module/user role *creation* is fine; it must not carry grants.
- **All grants — module roles are module-scoped (entities, microflows, pages):** the role in any `GRANT` must be declared *inside the same module as the element being granted*. This applies equally to entity access, microflow execute, and page view grants. Referencing `ModuleA.SomeRole` in a grant on any `ModuleB` element produces a CE error ("role not found in module"). Every module that owns grantable elements must declare its own module roles. User roles then *compose* module roles from all relevant modules:
  ```
  -- Each module declares its own roles
  create module role "ModuleB"."ReadRole";
  -- All three grant types use same-module roles only
  grant entity access on "ModuleB"."Widget" to "ModuleB"."ReadRole" (read: all);
  grant execute on microflow "ModuleB"."ACT_DoThing" to "ModuleB"."ReadRole";
  grant view on page "ModuleB"."Overview" to "ModuleB"."ReadRole";
  -- User roles compose across modules
  create user role "AppUser" ("System"."User", "ModuleA"."ReadRole", "ModuleB"."ReadRole");
  ```
  Anti-pattern to watch for: a single `security-setup.mdl` that uses a shared `Security.UserRole` to grant access to elements from three other modules — that will fail CE on every element outside `Security`.
- **Demo user passwords — never touch MxAdmin, never set passwords:** MxAdmin ships in every project with password `1`; do not wipe, reset, or re-create it. Demo users are created by name + role only — no password block. The user switches to a demo account inside the app from MxAdmin; no password is needed. Pattern: `create demo user "firstname.lastname" with roles "Module"."Role";` — nothing more.
- **`System.User` missing from user roles:** every user role must include `System.User` — without it, users cannot log in. mxbuild passes, SP loads clean, no CE errors — silent login failure only. Pattern: `create user role "MyRole" ("System"."User", "Module"."Role", ...)`. Applies to every user role, no exceptions.
- **Keyword collisions:** widget/element names must not match MDL/OQL reserved words (case-insensitive): `MIN`, `MAX`, `IN`, `OUT`, `COUNT`, `SUM`, `AVG`, `ROW`, `COLUMN`, `STATUS`, etc. Use descriptive names or double-quote them. Confirmed 2026-07-23 (TFC-TCXGraphPOC): a bare `Status` attribute name inside a `retrieve ... where` clause (`where ... and Status != Enum.Value`) compiled fine through `mxcli check` but failed a real `mx check` with CE0161 "Error(s) in XPath constraint" — quoting it (`"Status" != Enum.Value`) fixed it. This is consistent with the entity's own domain model already quoting `"Status"` specifically in its access-rule read lists (all its sibling attributes unquoted) — a signal worth checking for before assuming a reserved-word conflict is the cause of a CE0161.
- **`EXTENDS` placement:** goes **before** the opening paren: `CREATE PERSISTENT ENTITY Mod.Photo EXTENDS System.Image (...)` — not after.
- **No `CASE/WHEN`:** use nested `IF/ELSE`. No `TRY/CATCH`: use `ON ERROR { ... }`.
- **Enum in string context:** always `toString($Obj/EnumAttr)` — never use the enum attribute directly where a String is expected.
- **`$Obj/AttrName` not `$Obj/"AttrName"`:** quoting attribute names in path expressions passes `mxcli check` but produces CE0117 at mxbuild. Quotes are correct for identifiers in declarations; wrong for path traversal.
- **Cross-module association expression paths:** for navigation through a cross-module association to a nested attribute, the entity name must be included: `$Item/ModuleA.Item_Other/ModuleB.OtherEntity/Attr` not `$Item/ModuleA.Item_Other/Attr`. The short form silently compiles but SP rejects it (CE0117). A bare `= empty` check does NOT need the long form.
- **`mxcli check` does not catch corruption:** it validates MDL grammar only. Items 1–11 above are all semantically valid MDL that check passes — only mxbuild or SP open will reveal the damage. Always run the real mxbuild gate after exec (see `iterative-build-loop.md`).

## Final self-check (mandatory before reporting back)

After drafting a script, re-read it line by line against the table above — not just the mental pre-check you did before writing. Inline assoc-sets, conditional visibility inside datagrid customContent columns, and cross-module association traversals as widget datasources can creep in during drafting without intent. `mxcli check --references` will not catch them. This re-read is the actual corruption-prevention step.
