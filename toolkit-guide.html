mxcli Project Toolkit — how this works

You point this toolkit at legacy source code or a stack of requirements, and it interviews you through to a running Mendix app. Every stage produces an artifact you can show a customer, and every decision you make is recorded — nothing is silently decided for you.

1 · Three ways in — pick what you're starting from

Migration

Stages P, 0–7

You have legacy source code (OutSystems, Java/Angular, Node/React, …). Extractors read it, BRDs are generated from it, and the app is rebuilt in Mendix — with an optional data cutover at the end.

Requirements-driven

Stages P, 1–6

No legacy code — you have specs, BRDs, workshop outputs, wireframes, or an expert to interview. Document extraction and SME interviews replace the code extractors; everything after that is identical.

Greenfield

Stages P, 5–6

You already know what to build. Skip straight to the Mendix build discipline: gated module-by-module construction, then end-to-end testing.

Just the tools

No pipeline

You have an existing Mendix app and only want an audit, a lint pass, or a regression/e2e test net under it. Skip everything above — no intake, no stages, no gates. The agent grabs exactly the tools needed and hands you a findings report or a green test suite.

2 · The journey

✋ HARD GATE means the pipeline stops until you explicitly confirm a decision. SOFT GATE means unknowns get a recommended default, recorded as ASSUMED, and the run keeps moving.

  • P
    Kickoff SOFT GATE
    You're asked — where the source/requirements live, any licence constraints, whether an expert (SME) is available.
    You get — a scaffolded workspace: intake.md, PROJECT.md (your decision register), index.html (your live dashboard).
  • 0
    Triage ✋ HARD GATE MIGRATION ONLY
    You're asked — reuse an existing extraction pipeline or build a new one (with a coverage matrix as evidence); what to do about each missing dependency; what order to tackle a large source in.
    You get — assessment.md + triage.md: what's in your source, the risks, and a signed-off plan of attack.
  • 1
    Analysis SOFT GATE
    You're asked — do documents exist beyond the source folder? DB schema? Sample data? Who can answer what the code can't?
    You get — a knowledge base extracted three ways: from code, from documents, from the expert interview. Each path is done or explicitly declared unavailable — never silently skipped.
  • 2
    Requirements SOFT GATE
    You're asked — to confirm business rules the code or documents imply. Narrative is never invented.
    You get — validated BRDs (business requirements documents), one per module, checked clean.
  • 3
    Architecture & Design ✋ HARD GATE
    You're asked — module boundaries, buy-vs-build per gap, target security roles, data volumes, integration contracts, branding (logo/palette/type).
    You get — architecture diagrams, fit-gap analysis, a versioned design system, annotated wireframes.
  • 4
    Build Plan ✋ HARD GATE
    You're asked — acceptance criteria per module (what "done" means), target environments, iteration size.
    You get — a numbered, dependency-ordered build plan. Nothing gets built before you approve it.
  • 5
    Build SOFT GATE
    You're asked — to confirm each module's business-rule coverage checklist: the real definition of done, beyond "no errors".
    You get — working modules, one at a time, each passing an automatic snapshot → execute → verify → restore-on-failure loop. A bad write costs one batch, never the project.
  • 6
    Test SOFT GATE
    You're asked — which edge cases matter beyond the golden path.
    You get — Playwright walking your app as a real user, plus database assertions catching what the UI can't. Results reported verbatim.
  • 7
    Cutover ✋ HARD GATE MIGRATION ONLY
    You're asked — migrate legacy data, seed fresh, or drop it? Who flips the switch, when, and what's the rollback?
    You get — migration scripts and a cutover checklist. Runs only after testing passes — never before.

3 · How a gate works — you correct, you don't fill in forms

  1. The agent does its homework first. It never asks what it can derive from your source, the extraction, or the Mendix model itself.
  2. It proposes, with evidence. 2–4 concrete options, a recommendation, and why — citing your actual source (“your app has 4 auth roles, 3 are never checked; I recommend collapsing to 2”).
  3. It states its assumptions out loud so you can correct the premise, not just the conclusion.
  4. You answer in the terminal — multiple choice, “other” always available.
  5. The decision is written down twice: in that stage's HTML report and in PROJECT.md, marked CONFIRMED.
  6. Don't know? That's fine. The recommendation is applied and marked ASSUMED with the risk if wrong. A solo run never stalls — except at a ✋ gate, which waits for you.

4 · Three ways your app gets written — CLI, MCP+MDL, hand-rolled MCP

From Stage 5 on, the agent writes to your Mendix project (.mpr) through one of three routes. You don't have to pick — the agent checks a safety table (the "STOP table") before every write — but knowing the difference explains why it sometimes asks you to close or open Studio Pro.

CLI STUDIO PRO CLOSED

The bulk builder. mxcli exec script.mdl writes a whole reviewable script — entities, associations, microflows, security, navigation — to disk in one shot. Scaffolds an entire module at once, auto-snapshots before every run. Tradeoff: Studio Pro must be closed and reopened afterwards.

MCP + MDL STUDIO PRO OPEN

The live editor. mxcli --mcp exec routes the same MDL through Studio Pro's own engine while it's running — changes appear immediately, no restart. This is the mode for UI iteration: page layouts, widget wiring, visibility tweaks. It also sidesteps a class of disk-writer bugs.

Hand-rolled MCP STUDIO PRO OPEN

The precision tool. Raw JSON patches against Studio Pro's model API (pg_patch_page), for the few widget shapes MDL has no syntax for yet — complex datagrid columns, dropdown filters. Used only when the other two genuinely can't express the change.

In practice: CLI to build, MCP to refine. A typical module: one CLI exec scaffolds the domain model and logic → MCP+MDL polishes the pages → hand-rolled MCP only for the exceptions. Two operations never go through any of these — project settings and dropping secured attributes go to you, in the Studio Pro GUI, because they corrupt on every automated route. The agent will tell you when, with the exact click path.

5 · Where everything lands

<project-root>/               ← ONE folder per project — everything lives inside it
  PROJECT.md                  ← every decision, assumption, and open question
  intake.md · assessment.md · triage.md
  index.html                  ← your dashboard — open this anytime to see where you are
  source/                     ← your original source or requirements docs, read-only
  analysis/<source-name>/
    knowledge-base/           ← extraction JSON + BRDs
  architecture/ · design/     ← diagrams, fit-gap, design system, build plan
  <Project>.mpr               ← the target Mendix app

The toolkit clone (~/Mendix/mxcli-project-toolkit) stays elsewhere and stays clean —
nothing of yours ever lands in it. And analysis/ is INSIDE your project folder,
never a sibling next to it.

The dashboard regenerates from real project files every time a gate check runs (bin/gate-check.sh) — it can't drift from reality, because it's derived from it.

6 · Something looks broken? Don't panic — it's designed for this

“Studio Pro won't open / the project fails to load”

A bad write happened — and the toolkit snapshotted automatically before it. Tell the agent: The project won't load — restore the last snapshot. It runs bin/restore-mpr.sh, which restores both halves of the project together. You've lost at most one batch of changes.

“Studio Pro won't start / hangs on launch”

Almost always a stale background process from the last session. Tell the agent: SP won't start — kill any stale Studio Pro processes. That's the entire problem 95% of the time; a leftover .mpr.lock file is the other 5%.

Nothing here requires reinstalling anything or starting over. Git commits at every phase gate are the real safety history.
